### THREAD B — `Day of build: how Dayo's 6-digit code works`

**T1 (B/1)**
"Some students don't see the actual code — let me walk through how Dayo's 6-digit code onboarding is actually built.

Total: ~200 lines of TypeScript. ~3 days of work. ~$0/month of infra at current scale.

Here's the architecture."
[engagement: bookmark]

**T2 (B/2)**
"When Siddhant (me, the user) creates a timetable:
1. Generate a 6-digit numeric code.
2. Store a `batches` doc with: `code`, `creatorUid`, `createdAt`, `semester`.
3. Use the code as the key (sharded by last digit for evenly distributed reads).

Why last-digit sharding: avoids hot buckets during peak morning briefings."

**T3 (B/3)**
"When Siddhant's friend Rahul opens Dayo and types the code:
1. Read the `batches/<code>` doc.
2. Pull the `templateScheduleId`.
3. Clone that schedule into Rahul's user namespace.
4. Mark Rahul's batch as 'joined: <code>'.

No auth labyrinth. No 'who sent the invite.' No friend-graph."

**T4 (B/4)**
"Where things broke:

- The 6-digit namespace got 4 collisions on first deploy. Code now uses a probabilistic dedupe + retry.
- Some users lost the code mid-typing. Fixed: lead-zero-pad, single-line input.
- Firebase RTDB cost spiked when 50 students opened the briefing at exactly 7:00 AM. Moved to Firestore + cache layer."

**T5 (B/5)**
"The lesson:

Building viral loops isn't about clever tech. It's about designing the *minimum viable path* for someone to say 'try this' to a friend.

200 lines. One stat. 1,000 users.

Don't over-engineer."

### THREAD D — `What I learned about Indian college attendance`

**T1 (D/1)**
"When I started building Dayo I assumed attendance math was an engineering laziness problem.

After ~6 months of running the app, I now have a different take.

It's a sociology problem."
[engagement: bookmark]

**T2 (D/2)**
"What the math doesn't capture:

- Classrooms vary. Same subject, different teachers have wildly different 'Class actually happened' rates.
- Lab attendance is its own beast — most colleges enforce 100% lab attendance, not the 75% theory rule.
- Students in semester 5+ mentally write off semester 1 subjects. The math is correct; the engagement isn't."

**T3 (D/3)**
"What students actually want:

- A daily-answered 'should I go to college today?'
- Reassurance they're not the only one cutting class.
- A way to score attendance recovery without skipping the next two lectures following a fail."

**T4 (D/4)**
"The 'safe to bunk today' answer depends on:

- Current %
- Total classes held
- Target threshold
- Days until next assessment
- Class is theory vs lab vs tutorial

Dayo encodes all of these. Users get a clean answer every morning. Nobody else does, at least not in this country."

**T5 (D/5)**
"If you're building for college students, the rule is:

Don't build for the 75% rule. Build for the *question* the 75% rule forces every student to answer.

Build the answer, and your product wins."

## END MATTER

Use these 30 tweets as your starting inventory. Rotate the engagement tweets out for fresh ones every 2 weeks based on actual engagement data.

When (not if) a tweet takes off, follow up the next day with a thread that expands the same topic. The thread absorbs the audience from the viral first tweet.

Never reply to engagements on Saturday. Replies on Saturday get low impression credit. Tuesday and Wednesday 9–10 PM IST are the highest-CTR reply windows for Indian engineering-student audiences.
